  <action>
Replace every `docker exec` call in the 5 test scripts with the equivalent `docker compose exec` call. The migration rules are:

1. `docker exec claude-secure CMD` -> `docker compose exec claude CMD`
2. `docker exec claude-proxy CMD` -> `docker compose exec proxy CMD`
3. `docker exec claude-validator CMD` -> `docker compose exec validator CMD`
4. `docker exec -d claude-proxy CMD` -> `docker compose exec -d proxy CMD`
5. `docker exec -i claude-secure CMD` -> `docker compose exec -T claude CMD` (note: `-i` becomes `-T` for non-TTY in compose exec, which is correct for piping stdin)
6. `docker inspect claude-secure` -> `docker inspect $(docker compose ps -q claude)` (if any exist)

IMPORTANT: `docker compose exec` in non-interactive contexts (scripts) should use `-T` flag to disable TTY allocation. For commands that pipe stdin (like test-phase2.sh hook tests), use `docker compose exec -T`.

For test-phase2.sh specifically: all hook invocations use `docker exec -i claude-secure /etc/claude-secure/hooks/pre-tool-use.sh` with heredoc stdin. These become `docker compose exec -T claude /etc/claude-secure/hooks/pre-tool-use.sh`.

For test-phase3.sh: the `docker exec -d` (detached) calls become `docker compose exec -d`. The regular `docker exec claude-proxy` calls become `docker compose exec -T proxy`. The `docker exec claude-secure` calls become `docker compose exec -T claude`.

For test-phase1.sh: `docker exec claude-secure` -> `docker compose exec -T claude`, `docker exec claude-proxy` -> `docker compose exec -T proxy`.

For test-phase4.sh: single `docker exec claude-secure curl` -> `docker compose exec -T claude curl`.

For test-phase7.sh: `docker exec claude-proxy` -> `docker compose exec -T proxy`.

Do NOT modify test-phase6.sh (0 docker exec calls) or test-phase9.sh (0 docker exec calls -- it already uses docker compose commands).

The scripts must remain backward-compatible: when `COMPOSE_PROJECT_NAME` is the default project name, they still work. When it is `claude-test`, they also work. This is automatic since `docker compose exec` respects `COMPOSE_PROJECT_NAME` from the environment.
