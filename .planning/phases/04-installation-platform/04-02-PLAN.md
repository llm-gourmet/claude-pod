**Tests to implement:**

1. **INST-01: Dependency checker detects missing tools**
   - Source install.sh functions (use `source "$PROJECT_DIR/install.sh" --source-only` pattern, or extract the function and test it)
   - Simpler approach: Run `bash -c 'PATH=/nonexistent:$PATH; source install.sh 2>/dev/null; check_dependencies' 2>&1` and verify it reports missing tools
   - Alternative simpler approach: Just verify the function exists and the script parses: `bash -n "$PROJECT_DIR/install.sh"` and `grep -q 'check_dependencies' "$PROJECT_DIR/install.sh"`
   - Best approach for testability: Run the installer's check_dependencies with the real PATH (all deps should be present on the dev host). Verify exit 0. Then test with a modified PATH to hide `jq`: `env PATH=/usr/bin:/bin bash -c "... check_dependencies"` -- but this is fragile.
   - **Pragmatic approach:** Verify `bash -n install.sh` passes (syntax valid), verify all dependency names appear in the script, verify running `check_dependencies` on the current system succeeds (all deps present). Test the negative path by checking the script contains the error message pattern.

2. **INST-02 / PLAT-01 / PLAT-02 / PLAT-03: Platform detection**
   - Source install.sh and run `detect_platform` -- verify it sets `PLATFORM` to either "linux" or "wsl2"
   - Verify the output contains either "Detected platform: linux" or "Detected platform: wsl2"
   - Verify `iptables -V` output is logged (PLAT-03)

3. **INST-03: Auth setup writes .env**
   - Create a temp config dir, set `CONFIG_DIR` to it
   - Set `ANTHROPIC_API_KEY=test-key-12345` in env, run `setup_auth`
   - Verify `.env` file was created containing `ANTHROPIC_API_KEY=test-key-12345`
   - Verify `.env` has permissions 600: `stat -c '%a' "$tmpdir/.env"` equals "600"
   - Clean up temp dir

4. **INST-04: Directory structure and workspace**
   - Create a temp dir, set `CONFIG_DIR` to it
   - Run `setup_directories`, verify directory exists with permissions 700
   - Verify `stat -c '%a' "$tmpdir"` equals "700"

5. **INST-05: Docker images build**
   - Run `docker compose -f "$PROJECT_DIR/docker-compose.yml" config --quiet` -- verify exit 0 (compose file valid)
   - Run `docker compose -f "$PROJECT_DIR/docker-compose.yml" build` -- verify exit 0 (images build)
   - This is a real integration test that builds the images

6. **INST-06: CLI wrapper syntax and structure**
   - `bash -n "$PROJECT_DIR/bin/claude-secure"` passes
   - `grep -q 'docker compose down' "$PROJECT_DIR/bin/claude-secure"` (stop subcommand)
   - `grep -q 'docker compose ps' "$PROJECT_DIR/bin/claude-secure"` (status subcommand)
   - `grep -q 'COMPOSE_FILE' "$PROJECT_DIR/bin/claude-secure"` (works from any dir)

7. **PLAT-01/PLAT-02: Containers start and respond**
   - `cd "$PROJECT_DIR" && docker compose up -d` -- verify exit 0
   - Wait for containers: `docker compose ps --format json | jq -r '.State'` shows "running" for all 3
   - Verify proxy responds: `docker compose exec claude curl -s -o /dev/null -w '%{http_code}' http://proxy:8080/` (should get some response, even 400/405)
   - Actually, the claude container doesn't have curl. Use: `docker compose exec claude node -e "const http=require('http');http.get('http://proxy:8080',r=>{console.log(r.statusCode);r.resume()}).on('error',e=>console.log('ERR:'+e.message))"` -- or just check `docker compose ps` shows all running
   - Clean up: `docker compose down`

**Test structure:**
- Each test wrapped in a function or inline with clear comments
- Use subshells for tests that modify environment `( ... )` to avoid leaking state
- `report "INST-01" "Dependency checker syntax valid" "$result"` pattern
- Final summary: "Phase 4: $PASS/$TOTAL passed" and exit 1 if FAIL > 0

Actually, since install.sh may not have the source guard (Plan 01 doesn't explicitly add one), add it: Modify install.sh to wrap the final `main "$@"` in:
