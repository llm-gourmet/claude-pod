# Quick Task 260415-e51: Audit and Fix install.sh setup_auth Empty-.env Bug Summary

Hardened `install.sh setup_auth` and `bin/claude-secure setup_profile_auth` against empty/non-TTY/newline-tainted credential input, marked `install.sh` executable in the git index, and corrected README inaccuracies about where the installer writes files on the host.

## What Changed

### Task 1 ŌĆö Harden install.sh setup_auth (commit 5eb083c)

Replaced the four-echo writer pattern in `setup_auth` (install.sh lines 236-291) with:

1. **Unified `write_env_file` helper** (nested in `setup_auth`): rejects empty values and newline/CR-tainted values with `log_error` + `exit 1`, then writes the entire four-line `.env` in a single grouped redirection (`{ printf...; } > "$env_file"`) so a mid-write abort can never leave a truncated file on disk.
2. **TTY guard** (`[ ! -t 0 ]`): before the interactive menu, aborts with an actionable error message teaching the user to use `sudo -E CLAUDE_CODE_OAUTH_TOKEN="$CLAUDE_CODE_OAUTH_TOKEN" bash install.sh` when stdin is not a terminal. This closes the root-cause path where `sudo` scrubs the environment and the interactive prompt silently consumes EOF.
3. **Non-empty validation on typed input**: after each `read -rsp`, checks `[ -z "$token" ]` / `[ -z "$key" ]` and aborts before touching the filesystem. An empty prompt response no longer produces a header-only `.env`.
4. **Removed the dangling `chmod 600 "$env_file"` at the old line 290** ŌĆö `chmod` now lives inside `write_env_file`, which only runs when a real value was written.

The two non-interactive env-var branches (`ANTHROPIC_API_KEY`/`CLAUDE_CODE_OAUTH_TOKEN` already set) now route through the same helper, so they get the same newline validation as interactive input.

### Task 2 ŌĆö Mirror hardening in bin/claude-secure, fix install.sh exec bit, correct README (commit 4718226)

**bin/claude-secure setup_profile_auth:** applied the same four hardening steps as Task 1 ŌĆö TTY guard, empty-value guards on both `$key` and `$token`, newline/CR case-statement rejection, and a single grouped redirection for the four-line `.env`. Error reporting uses `echo "ERROR: ..." >&2; return 1` (not `exit 1`) because this function runs inside the CLI wrapper, not the installer.

**create_profile tmpdir cleanup:** registered the function's `$tmpdir` with the existing `_CLEANUP_FILES` array (line 302). The project already has a global `trap cleanup EXIT` (line 55) that walks `_CLEANUP_FILES` ŌĆö this is used 14 times elsewhere in the file. Now a failed `setup_profile_auth` no longer leaves a half-built profile directory behind.

**install.sh executable bit:** ran `git update-index --chmod=+x install.sh` and `chmod +x install.sh`. Git now tracks mode `100755` instead of `100644`, so `sudo ./install.sh` works on a fresh clone without a separate `chmod +x` step. The commit records a mode change line: `mode change 100644 => 100755 install.sh`.

**README.md ŌĆö three surgical edits:**

1. Installation section: replaced "writes security hooks to `/etc/claude-secure/`" with an accurate description that says the installer builds Docker images (PreToolUse hook baked into the image at build time from `claude/hooks/`), installs the CLI, copies the project tree to `~/.claude-secure/app/`, and writes a default profile. Added a non-interactive install snippet documenting `sudo -E CLAUDE_CODE_OAUTH_TOKEN="$CLAUDE_CODE_OAUTH_TOKEN" ./install.sh`.

2. Architecture > PreToolUse hook section: added **"inside the Claude container"** and the baked-at-build-time parenthetical to the hook-path sentence so readers cannot mistake `/etc/claude-secure/hooks/pre-tool-use.sh` for a host path.

3. First-run prompts list: expanded to mention that the auth choice is interactive and to cross-reference the `sudo -E` escape hatch for non-interactive installs.

No other README sections were touched.

## Deviations from Plan

None that required a decision. Two minor naming reconciliations the plan's language didn't exactly match the code:

- Plan refers to `do_create_profile` in bin/claude-secure; the actual function name is `create_profile`. Applied the tmpdir-cleanup change to `create_profile` (lines 295-354). The plan's description of the function body matched the real `create_profile` exactly.
- Plan suggests "add one single `trap 'rm -rf "$tmpdir"' EXIT RETURN` inside `do_create_profile`". Instead of adding a new trap, registered `$tmpdir` with the project's existing `_CLEANUP_FILES` + global `trap cleanup EXIT` pattern (documented in the Decisions frontmatter). Rationale: bash does not support proper function-scoped traps, the project already uses `_CLEANUP_FILES` in 14 other places, and this keeps the cleanup mechanism uniform with the rest of the file. Net behaviour is identical ŌĆö cleaner and consistent with project conventions.

Both are Rule-3-style adjustments (naming/pattern fit the existing codebase) and were made without user input.
